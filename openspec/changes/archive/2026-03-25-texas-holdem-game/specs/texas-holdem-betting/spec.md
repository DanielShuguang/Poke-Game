## 新增需求

### 需求：基础投注动作
系统必须支持 5 种标准投注动作：Fold（弃牌）、Check（过牌）、Call（跟注）、Raise（加注）、All-in（全押）。系统禁止在不符合当前游戏状态的情况下执行动作（如无人下注时 Call）。

#### 场景：Fold
- **当** 玩家选择 Fold
- **那么** 玩家退出本局，其已投入筹码留在底池，不再参与后续行动和摊牌

#### 场景：Check
- **当** 当前轮次无人下注且玩家选择 Check
- **那么** 行动权传递给下一位玩家，底池不变

#### 场景：Call
- **当** 当前轮次有人下注且玩家选择 Call
- **那么** 玩家补足至当前最高注额，筹码从玩家扣除并加入底池

#### 场景：Raise
- **当** 玩家选择 Raise 并输入加注金额
- **那么** 系统验证金额不低于最小加注额（上一次加注增量），扣除玩家筹码，其他玩家需重新决策

#### 场景：All-in
- **当** 玩家筹码不足以完成 Call 或主动选择 All-in
- **那么** 玩家押入全部剩余筹码，触发边池计算流程

### 需求：边池（Side Pot）计算
系统必须在存在 All-in 玩家时正确拆分主池（Main Pot）和边池（Side Pot），确保每位玩家只能赢取其有资格参与的池。

#### 场景：单个 All-in 玩家建立边池
- **当** 1名玩家 All-in，其筹码少于其他玩家的投注额
- **那么** 主池上限为 All-in 金额 × 参与人数，超出部分形成边池，All-in 玩家不参与边池

#### 场景：多个 All-in 玩家多级边池
- **当** 多名玩家以不同金额 All-in
- **那么** 系统按金额从小到大逐级拆分，形成多个边池，每个边池有各自的合格参与者列表

#### 场景：平分底池（Split Pot）
- **当** 多名玩家摊牌后牌型完全相同
- **那么** 底池平均分配；若筹码无法整除，余数归 Small Blind 位玩家（或最靠近庄家左侧的活跃玩家）

### 需求：最小加注限制
系统必须强制执行最小加注规则：每次 Raise 金额不得低于上一次加注的增量（No-Limit Hold'em 标准）。

#### 场景：最小加注验证
- **当** 玩家输入 Raise 金额
- **那么** 系统验证增量 ≥ 上一次 Raise 增量；若不足则提示并拒绝

#### 场景：首次下注最小额
- **当** 本轮无人下注，玩家首次下注
- **那么** 最小下注额为 1 个大盲注

### 需求：筹码结算
系统必须在每局结束（提前结束或 Showdown 后）将各底池筹码分配给获胜玩家，且全局筹码总量必须守恒。

#### 场景：单一赢家结算
- **当** Showdown 后确定唯一最佳牌型玩家
- **那么** 该玩家获得主池全部筹码，其筹码余额相应增加

#### 场景：边池赢家结算
- **当** Showdown 后含 All-in 场景
- **那么** 主池由资格最广的最佳牌型玩家获得，各边池由对应资格范围内最佳牌型玩家获得
