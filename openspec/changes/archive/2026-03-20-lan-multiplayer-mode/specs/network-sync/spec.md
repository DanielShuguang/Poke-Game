# 网络同步规范

## 新增需求

### 需求：游戏状态必须实时同步
系统必须确保所有玩家的游戏状态（牌面、出牌、回合）在 100ms 内同步。

#### 场景：发牌同步
- **当** 游戏开始，房主执行发牌操作
- **那么** 系统必须在 50ms 内将每位玩家的手牌通过 WebSocket 发送给对应玩家

#### 场景：出牌同步
- **当** 玩家 A 出牌（如"三带一"）
- **那么** 系统必须在 100ms 内向所有玩家广播出牌信息

#### 场景：叫地主同步
- **当** 玩家 B 叫地主
- **那么** 系统必须在 100ms 内向所有玩家广播叫地主结果

### 需求：系统必须使用事件溯源机制
系统必须通过事件日志记录所有游戏操作，支持断线重连恢复。

#### 场景：事件日志格式
- **当** 发生游戏操作
- **那么** 系统必须生成事件对象，包含：事件 ID、类型、载荷、时间戳

#### 场景：事件 ID 唯一性
- **当** 生成新事件
- **那么** 系统必须使用 UUID 或递增序列号确保事件 ID 唯一

### 需求：房主必须验证操作合法性
系统必须由房主验证所有玩家操作，拒绝非法操作。

#### 场景：出牌验证
- **当** 玩家 A 发送出牌请求（如单张 3）
- **那么** 房主必须验证：玩家是否拥有该牌、是否符合出牌规则、是否轮到该玩家

#### 场景：拒绝非法操作
- **当** 玩家 A 尝试出牌"王炸"但手牌中没有
- **那么** 房主必须拒绝操作，返回错误消息"非法出牌：未拥有这些牌"

### 需求：客户端必须缓存事件日志
系统必须确保客户端缓存至少 100 条事件日志，支持快速恢复。

#### 场景：断线重连恢复
- **当** 玩家 A 网络中断后重连，发送 `lastEventId = "evt_42"`
- **那么** 房主必须返回事件 ID 43 及之后的所有事件

#### 场景：事件日志超限
- **当** 事件日志超过 100 条
- **那么** 系统必须删除最早的事件，保持队列长度为 100

### 需求：系统必须处理网络延迟
系统必须优化消息大小和发送频率，降低网络延迟影响。

#### 场景：消息压缩
- **当** 发送事件载荷超过 1KB
- **那么** 系统必须使用 gzip 压缩消息

#### 场景：批量发送
- **当** 短时间内发生多个事件（如连续出牌）
- **那么** 系统必须合并为批量消息发送，减少网络请求次数

### 需求：系统必须支持心跳检测
系统必须通过心跳机制检测玩家在线状态。

#### 场景：心跳间隔
- **当** 游戏进行中
- **那么** 系统必须每 5 秒发送一次心跳包（Ping/Pong）

#### 场景：心跳超时
- **当** 玩家连续 3 次心跳未响应（15 秒）
- **那么** 系统必须标记玩家为"离线"，触发离线处理逻辑
